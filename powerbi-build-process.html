Phase 1 — Connect & Prepare the Data

Before a single visual is built, the data model must be correct. This phase covers importing the Excel dataset, fixing column headers and data types in Power Query, and defining the 6 relationships that allow all 7 tables to cross-filter. Several real problems are documented here — duplicate header rows, the relationship direction error, and a hidden OK button that required a keyboard workaround.

1
Open Power BI Desktop
Setup

Launch Power BI Desktop and close the splash screen. Confirm three icons are visible on the left sidebar:

  • Report (bar chart icon) — where all visuals are built. Start here.
  • Data (table icon) — browse raw imported data.
  • Model (diagram icon) — define relationships between tables.
Don't have Power BI Desktop? Download it free from microsoft.com/en-us/power-bi/desktop. Windows only. Mac users need a Windows virtual machine.

3
Fix Headers & Column Types in Power Query
Troubleshoot
⚠ Real Problem — Date column threw an error immediately

Attempting to change the Date column type caused an error: column not found. The dataset has a coloured title banner row (e.g. "DEPARTMENT PERFORMANCE — DAILY…") sitting above the real column headers. Power Query treated the banner as row 1, so "Date" did not yet exist as a column name.

Fix — apply these steps in this exact order for each table:

  • Home → Remove Top Rows → 1 → OK — removes the title banner row (Daily_KPIs only needed Use First Row as Headers, as it had one fewer extra row)
  • Home → Use First Row as Headers — promotes the real column names (Date, Department etc.) to headers
  • Now set column types by clicking the icon left of each column header
ColumnTypeNote
DateDateKey join field — must be Date not Date/Time
Department, Day of Week, Quarter, Day Type, On LeaveTextCategorical fields
Week No.Whole NumberISO week 1–53
All columns ending ($)Fixed Decimal NumberRevenue, cost columns
All columns ending (%)Decimal NumberStored as 0.0–1.0 decimals

When all 7 tables are fixed: Home → Close & Apply. Power BI loads all 20,130 rows.

4
Define the 6 Relationships in Model View
Troubleshoot

Click the Model icon on the left sidebar. Go to Home → Manage Relationships → New for each relationship. All 6 connect through the Date field.

⚠ Real Problem 1 — Duplicate value error

Setting Daily_KPIs on the From side caused:

Column "Date" in Table 'Department_Daily' contains a duplicate value '1/3/2024' and this is not allowed for columns on the one side of a many-to-one relationship.

Why: Department_Daily has 8 rows per date (one per department), so its Date is not unique. The "one" side of a many-to-one must be unique. Daily_KPIs has exactly one row per date — it belongs on the To (one) side.

Fix: Detail table always in the From field. Daily_KPIs always in the To field.

From (many side)To (one side)
Department_Daily [Date]Daily_KPIs [Date]
Physician_Daily [Date]Daily_KPIs [Date]
Patient_Outcomes_Daily [Date]Daily_KPIs [Date]
Financial_Daily [Date]Daily_KPIs [Date]
Equipment_Daily [Date]Daily_KPIs [Date]
Staffing_Daily [Date]Daily_KPIs [Date]

In each dialog also set: Cardinality → Many to one (*:1) · ✅ Make this relationship active

⚠ Real Problem 2 — OK button not visible on screen

After filling in all settings, the OK / Save / Confirm button was not visible — it was rendered below the bottom of the screen, hidden behind the Windows taskbar. Scrolling, resizing, and moving the window all failed to reveal it.

Fix: Press Tab repeatedly to cycle through all interactive elements in the dialog, then press Enter once the focus reaches the confirm button. This saves the relationship without needing to click it.

⚠ Real Problem 3 — Security filter error on 2nd relationship
Table 'Daily_KPIs' already has a relationship where Security Filtering Behavior is set to Both. Only one relationship per table with this setting is allowed.

Fix: Leave "Apply security filters in both directions" unchecked for all 6 relationships. This setting relates to row-level security and is not needed for this dashboard.

Model View: 7 tables connected via Date. Daily_KPIs is the central fact table on the "one" side of all 6 relationships. The _Measures table and Data_Dictionary are unrelated lookup tables.

Phase 2 — Build the Measures Library

All DAX measures are written before any visuals are built. They live in a single _Measures table, separating logic from source data. One measure required a formula correction during the build — the HAI Rate was returning 0.000% until the formula was changed to reference the pre-calculated column directly.

5
Create the _Measures Table
Setup

Home ribbon → Enter Data. Name the table _Measures. Add a column named Placeholder with value 0. Click Load.

To add a measure: click _Measures in the Data pane → Table Tools → New Measure → type the formula → press Enter → right-click the measure to set its number format.

The underscore prefix in _Measures forces it to sort above all source tables in the Data pane, making it easy to find. This is a standard professional convention in Power BI development.

Power BI defaulted to Auto display units, shortening 21,473 to "21k". Fix: Format visual pane → Callout value → Display units → change from Auto to None.

⚠ Real Problem — All department bars showing the same value (21,473)

Using Total Admissions on the Department bar chart made every bar show 21,473 — the hospital-wide total. The measure has no department context because it sums from Daily_KPIs which has one row per day (not per department).

Fix: Dept Total Admissions sums from Department_Daily which has one row per department per day — giving correctly different values per department (Emergency ~4,685, Orthopedics ~2,608 etc.).

-- Corrected formula — see troubleshoot below
HAI Rate % = AVERAGE ( Patient_Outcomes_Daily[HAI Rate (%)] )
⚠ Real Problem — HAI Rate % showing 0.000%

The original formula was:

HAI Rate % = DIVIDE ( SUM ( Patient_Outcomes_Daily[HAI Cases] ), SUM ( Patient_Outcomes_Daily[Cases] ), 0 )

Investigation: The HAI Rate (%) column already contains the pre-calculated rate as a decimal (0.0075, 0.009 etc.). The original formula was dividing a small HAI count by a large total case count — producing an even smaller decimal that showed as 0.000% even with 2 decimal formatting.

Fix: AVERAGE the pre-calculated column directly. No division needed — the work is already done. The corrected measure returns ~0.9% as expected.

Chart Y-axis also needed fixing: Default 0–100% scale made the 0.7–1.1% variation invisible as a flat line. Fix: Format visual pane → Y-axis → Range → Minimum: 0.005, Maximum: 0.015. Set the constant target line to 0.01.

⚠ Real Problem — Months out of order, no Month column in dataset

Two approaches failed: creating a Month No column and sorting by it had no effect; creating a Month Label column with FORMAT() and Sort by Column also failed to order correctly in the chart.

Working fix: Drag the Date field directly into the X-axis. Power BI creates a date hierarchy automatically. Click the drill-down arrow on the chart until it shows the Month level. The months sort Jan–Dec correctly because they are driven by underlying date values, not text labels.

Chart 2 — Revenue by Department (horizontal bar): Y-axis: Department. X-axis: Dept Total Revenue. Sort descending. Bar colour: #9D174D. ICU at top ~$20.8M.

Chart 4 — Admissions by Day of Week (column): X-axis: Day of Week. Y-axis: Total Admissions.

⚠ Real Problem — Days in alphabetical order (Friday, Monday, Saturday…)

Power BI sorted Day of Week alphabetically by default. Fix: create a numeric sort column on Daily_KPIs:

Day No = WEEKDAY ( Daily_KPIs[Date], 2 )

Then: click Day of Week column → Column Tools → Sort by Column → Day No. On the chart: ··· → Sort by → Day No → Sort ascending. Saturday and Sunday show visibly lower bars than weekdays.

Phase 5 — Sync Slicers, Validate & Publish

Connect all 6 pages via synced slicers, validate FY 2024 totals against known benchmarks, then save and export. Without slicer sync, each page filters independently — what looks like a dashboard is actually 6 separate reports.

21
Sync Slicers Across All Pages
Setup

Click the Date slicer on Page 1. Go to View ribbon → Sync Slicers. A panel opens showing all 6 pages. Tick both columns (sync + visible) for all 6 pages.

Go to each of Pages 2, 4, and 6. Click the Department slicer on each → Sync Slicers → tick both columns for Pages 2, 4, and 6 only. Leave Clinical Quality and Physician Productivity unticked for the Department slicer — those pages have their own category-specific slicers.

The sync column (chain icon) means the slicer reacts to changes elsewhere. The visible column (eye icon) means a slicer control appears physically on that page. Tick both for a fully connected experience where the user can filter from any page.
